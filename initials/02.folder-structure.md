Project Folder Structure: Indecisive App

For a hackathon, a "monorepo" approach (keeping both frontend and backend in one main repository) is usually the easiest to manage and deploy.

Here is the recommended folder structure based on the Google ADK and Vite/React stack.

```plaintext
apps-hackathon-genai-apac/
├── backend/                       # Python API (Deployed to Cloud Run)
│   ├── app/
│   │   ├── __init__.py
│   │   ├── main.py                # FastAPI entry point (handles routing)
│   │   ├── agents/                # Google ADK Agent definitions
│   │   │   ├── __init__.py
│   │   │   ├── primary.py         # Supervisor agent logic
│   │   │   ├── interviewer.py     # Sub-agent
│   │   │   ├── researcher.py      # Sub-agent (Google Search Grounding)
│   │   │   ├── evaluator.py       # Sub-agent
│   │   │   └── supporter.py       # Sub-agent
│   │   ├── mcp/                   # Model Context Protocol integrations
│   │   │   ├── __init__.py
│   │   │   ├── sqlite_client.py   # Connects to SQLite MCP
│   │   │   └── notion_client.py   # Connects to Notion/Drive MCP
│   │   ├── core/                  # Configurations and database
│   │   │   ├── __init__.py
│   │   │   ├── config.py          # Environment variables (API keys)
│   │   │   └── firestore.py       # Google Cloud Firestore setup
│   │   ├── api/                   # API Routes
│   │   │   ├── __init__.py
│   │   │   └── chat.py            # Endpoints for the frontend to call
│   │   └── models/                # Pydantic schemas for data validation
│   │       ├── __init__.py
│   │       └── schemas.py         # Request/Response definitions
│   ├── requirements.txt           # Python dependencies (google-adk, fastapi, etc.)
│   └── Dockerfile                 # Instructions to build the Cloud Run container
│
├── frontend/                      # Vite + React App (Deployed to Vercel/Firebase)
│   ├── public/
│   ├── src/
│   │   ├── assets/
│   │   ├── components/            # Reusable UI elements
│   │   │   ├── ChatInterface.jsx  # Main chat window
│   │   │   ├── DecisionMatrix.jsx # UI to display the SQLite evaluated options
│   │   │   └── MessageBubble.jsx
│   │   ├── services/              # API connection logic
│   │   │   └── api.js             # Axios/Fetch calls to the Python backend
│   │   ├── App.jsx                # Main React component
│   │   └── main.jsx               # React entry point
│   ├── index.html
│   ├── package.json               # Node.js dependencies
│   ├── tailwind.config.js         # Styling configuration
│   └── vite.config.js             # Bundler configuration
│
├── initials/                      # Documents and notes for the hackathon
├── .gitignore
└── README.md                      # Documentation for judges
```

Key Benefits of this Structure:
- Separation of Concerns: The agents folder isolates your Google ADK logic from your standard web server logic (api), making it much easier to debug LLM prompts without breaking your web server.
- MCP Isolation: The mcp folder keeps external tool connections clean. If an MCP server fails, it's easy to mock the response here during the hackathon.
- Cloud Run Ready: The Dockerfile sits exactly where it needs to be in the backend directory to easily build and deploy the API to GCP.
